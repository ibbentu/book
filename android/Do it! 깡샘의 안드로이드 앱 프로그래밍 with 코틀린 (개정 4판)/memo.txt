#배포 파일
APK / AAB
APK = Android application package
- 전통적인 배포 파일로 컴파일된 코드와 리소스를 묶어서 키로 서명한 파일
ABB = Android app bundle
- ABB 파일을 Play스토어에 등록하면 사용자 기기에 맞게 최적화된 APK 파일을 Play스토어가 생성

#앱서명
개발자
- 개발자가 만든 키 1개로 앱을 관리
- 앱을 업데이트하기 위해서는 이전 버전과 동일한 키로 서명해야하지만 분실하거나 도용되었을 때의 대처가 어려움
- 새로운 키를 만들어 서명하면 새로운 앱으로 등록됨
구글Play
- 개발자가 관리하는 업로드 키, 구글Play가 만드는 앱 서명 키 2개로 키를 관리함
- 업로드 키는 구글Play에 등록할 때 사용
- 앱 서명 키는 구글Play가 업로드된 앱을 서명해 사용자에게 전달할 때 사용
- 키가 구분되어있어 업로드 키가 분실되거나 도용되어도 앱을 계속 업데이트 할 수 있다

#구글Play스토어 게시
준비해야할 파일
- AAB 파일: 앱을 서명한 배포 파일
- 앱 아이콘 이미지: 512*512px 크기와 1MB미만의 JPEG 혹은 PNG 파일
- 그래픽 이미지: 프로모션 때 사용할 이미지. 1024*500px, 15MB 미만의 JPEG 혹은 24비트 PNG 파일
- 스크린샷: 실제 실행한 스크린샷. 320~3840px 크기(비율16:9, 9:16)와 개당 8MB미만의 JPEG 혹은 24비트 PNG 이미지 파일 2~8장
- 태블릿 스크린샷: 7인치 혹은 10인치 태블릿 스크린샷. 320~3840px 크기(비율16:9, 9:16)와 개당 8MB미만의 JPEG 혹은 24비트 PNG 이미지 파일 2~8장

2장 - 안드로이드 앱의 기본 구조

안드로이드의 특징
- 리눅스 기반
- Java나 Kotlin을 사용해 개발

ART(Android runtime)
- 앱을 실행하는 역할
- Java 코드를 DEX파일로 컴파일 한후 ART가 DEX파일을 실행한다

안드로이드 NDK(native development kit)
- C/C++ 라이브러리를 이용

안드로이드 버전
- 개발할 때 사용하는 버전은 API레벨(SDK버전)이다

#안드로이드 앱 개발의 특징
컴포넌트를 기반으로 한 개발
- 하나의 앱은 여러 컴포넌트로 구성된다.
- 안드로이드는 클래스로 컴포넌트를 개발한다.
(대략적으로 하나의 클래스가 하나의 컴포넌트가 된다)
- 클래스에는 컴포넌트 클래스와 일반 클래스가 있다.
- 일반 클래스는 클래스의 객체 생성부터 소멸까지 생명주기 관리를 개발자 코드에서 한다.
- 컴포넌트 클래스는 클래스의 생명 주기를 안드로이드 시스템에서 관리한다.

4종의 안드로이드 컴포넌트
- 액티비티: 화면을 구성하는 컴포넌트. 출력을 담당한다.
- 서비스: 백그라운드 작업을 하는 컴포넌트. 화면과 관계없이 장시간 실행되어야하는 작업을 담당한다.
- 콘텐츠 프로바이더: 앱의 데이터를 공유하는 컴포넌트. 다른 앱들과의 데이터 공유를 담당한다.
- 브로드캐스트 리시버: 시스템 이벤트가 발생할 때 실행이 되도록 만드는 컴포넌트. 시스템 이벤트는 부팅 완료, 배터리 방전 등과 같은 시스템에서 발생하는 이벤트를 뜻한다.

4종의 안드로이드 컴포넌트 구분
- 상속받은 클래스의 상위 클래스로 구분하면 된다.

컴포넌트는 앱 내부에서 독립된 실행 단위이다.
- 컴포넌트끼리 서로 종속되지 않아 코드 결합이 발생되지 않는다.
- 컴포넌트의 생명주기를 안드로이드 시스템에서 관리하기 때문에 직접 객체를 생성해 실행 할 수 없다.
- 안드로이드 시스템을 통해 클래스들이 서로 종속되지 않고 독립적으로 실행된다.

앱 실행 시점이 다양하다.
- 컴포넌트가 독립적으로 실행되는 특징으로 실행 시점이 다양할 수 있다.

애플리케이션 라이브러리를 사용할 수 있다.
- 다른 애플리케이션을 라이브러리처럼 이용할 수 있다.

리소스를 활용한 개발
- 리소스란 코드에서 정적인 값을 분리한 것이다.
- 대표적으로 문자열을 리소스로 등록해 활용하는 것이다.

# 앱 구성 파일 분석
프로젝트 폴더 구성 (app > src > main)
- 프로젝트 생성 시 app이라는 모듈이 자동으로 생성된다
- 일반적으로 하나의 모듈은 하나의 앱을 의미하고 프로젝트는 여러 모듈을 묶어서 관리하는 것을 의미한다

모듈 폴더 구성
- build.gradle.kts = 빌드 설정 파일
- AndroidManifest.xml = 앱의 메인 환경 파일
- res = 리소스 폴더
- activity_main.xml = 레이아웃 XML 파일
- MainActivity.kt = 메인 엑티비티 파일

Gradle 빌드 설정 파일
- 대부분의 빌드 설정은 모듈의 gradle 파일에 작성한다
- 중요한 설정
plugins - 기본적으로 android.application, kotlin.android 가 선언되어있다
compileSdk = 앱을 컴파일하거나 빌드할 때 적용할 버전 설정
applicationId = 앱의 식별자 설정
minSdk = 앱을 설치할 수 있는 기기의 최소 SDK버전
targetSdk = 개발 시 사용할 SDK버전
versionCode, versionName = 앱의 버전
compileOptions, kotlinOptions = 개발 언어의 버전 설정
dependencies = 라이브러리 선언

메인 환경 파일 - AndroidManifest.xml
<manifest> = 루트 태그
<application> = 앱 전체를 대상으로 하는 설정. xml값이 @으로 시작하면 리소스를 의미한다.
- icon = 실행 아이콘
- label = 앱의 이름
- theme = 테마 설정
컴포넌트는 manifest파일에 등록해야 시스템이 인식한다.
<activity> = 액티비티 태그
<service> = 서비스 태그
<receiver> = 브로드캐스트 리시버 태그
<provider> = 콘텐츠 프로바이더 태그
<intent-filter> = 생략 가능 태그
-> 내부 <action>태그의 name값이 android.intent.action.MAIN,
        <category>태그의 name값이 android.intent.category.LAUNCHER 로 설정되어있다면
        앱 아이콘을 선택했을 때 실행되는 액티비티라는 의미이다.
-> 만약 생략되고 다른 부분에도 없다면 앱을 설치해도 앱 아이콘이 생성되지 않아 일반적으로 사용자가 직접 실행할 수 없다.

리소스 폴더
- res폴더는 앱의 리소스를 등록하는 목적으로 사용된다.
- 자동으로 생성되는 폴더는 다음과 같다.
    - drawable: 이미지
    - layout: UI 구성에 필요한 XML 리소스
    - mipmap: 앱 아이콘 이미지
    - values: 문자열 등의 값으로 이용되는 리소스
- 리소스들은 R.java파일에 상수 변수로 등록되며 코드에서 상수 변수로 리소스를 이용한다.
- res하위의 폴더명은 지정된 폴더명을 사용해야 한다.
- 각 리소스 폴더에 하위 폴더를 정의할 수 없다.
- 리소스 파일명은 Java의 이름 규칙을 위배할 수 없다.
- 리소스 파일명에 대문자를 이용할 수 없다.

레이아웃 XML 파일 = 화면을 구성하는 파일
메인 액티비티 파일 = manifest 파일의 설정값에 따라 앱 아이콘 실행 시 MainActivity.kt 파일이 실행된다.

3장 코틀린
코틀린은 Java의 가상 머신인 JVM에 기반을 둔 언어이기 때문에 JVM에서도 실행이 가능하다.
코틀린의 .kt 파일을 컴파일하면 자바 바이트 코드가 생성된다.

Kotlin의 장점
- 표현력과 간결함
- 안전한 코드: 널 안정성(null safety)을 지원한다.
- 상호 운용성: Kotlin과 Java는 호환이 되기 때문에 혼용해서 사용해도 된다.
- 구조화 동시성: 코루틴으로 비동기 프로그래밍을 간소화할 수 있다.
